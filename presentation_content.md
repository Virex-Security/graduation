# العرض التقديمي: نظام Virex الأمني (API Security & SIEM Dashboard)

---

## 1. ملخص النظام (System Overview)
**الغرض الأساسي:**
نظام **Virex** هو منصة أمنية متكاملة (Security Information and Event Management - SIEM) مصممة لحماية واجهات برمجة التطبيقات (APIs) من الهجمات السيبرانية الشائعة والمتقدمة، مع توفير لوحة تحكم تفاعلية لمراقبة التهديدات في الوقت الفعلي.

**المشكلة التي يعالجها:**
مع تزايد الهجمات على تطبيقات الويب، تعجز أنظمة الحماية التقليدية (WAF) المعتمدة على القواعد الثابتة عن اكتشاف الهجمات المعقدة أو المشفرة (Zero-day attacks). يحل النظام هذه المشكلة عبر تقديم "محرك فحص هجين" يجمع بين القواعد الصارمة والذكاء الاصطناعي لاكتشاف التهديدات بدقة وتقليل الإيجابيات الخاطئة (False Positives)، بالإضافة إلى أتمتة حظر المخترقين.

---

## 2. الهيكل التقني (Technical Architecture)
يعتمد النظام على معمارية الخدمات المصغرة (Microservices-like) لضمان الفصل بين بيئة الحماية وبيئة المراقبة:

- **الواجهة الخلفية (Backend & API Service):** 
  - مبرمجة بلغة **Python** باستخدام إطار عمل **Flask** (تعمل على المنفذ 5000).
  - تتضمن خط أنابيب (Pipeline) لمعالجة الطلبات: فلترة الطلبات التافهة > فحص معدل الطلبات (Rate Limiting) > كشف الثغرات (WAF) > تحليل الذكاء الاصطناعي.
- **لوحة التحكم (SIEM Dashboard):**
  - مبرمجة بـ **Flask Templates** مع **HTML/CSS/JS** (تعمل بشكل مستقل على المنفذ 8070).
  - تتلقى التنبيهات في الوقت الفعلي من الـ API وتقوم بعرض الإحصائيات وإدارة الحوادث.
- **محرك الحماية والذكاء الاصطناعي (Security Engine):**
  - **الطبقة الأولى:** WAF يعتمد على التعابير النمطية (Regex) لاكتشاف (SQLi, XSS, Path Traversal, CSRF, SSRF).
  - **الطبقة الثانية:** نموذج تعلم آلي (Machine Learning) مبني باستخدام مكتبة **scikit-learn** يعتمد على (TF-IDF + Random Forest) لتحليل النصوص واكتشاف الشذوذ الأمني وإعطاء نسبة خطورة (Risk Score).

---

## 3. تحليل البيانات (Database Schema & ERD Analysis)
يستخدم النظام قاعدة بيانات **SQLite** مهيأة بنظام (WAL) لدعم القراءة والكتابة المتزامنة. تتكون الجداول الأساسية من:

1. **إدارة المستخدمين والصلاحيات (Users & Roles):**
   - جداول `users`, `roles`, `departments`.
   - **العلاقة:** علاقة (1-to-M) بين الصلاحيات والمستخدمين لتطبيق (RBAC - Role Based Access Control) مثل (Admin, Analyst, User).
2. **قواعد الحماية (WAF Rules):**
   - جدول `rules`: يخزن التعابير النمطية (Regex patterns) وأنواع الهجمات وحالة تفعيل القاعدة، مما يتيح تحديث قواعد الحماية ديناميكياً دون إعادة تشغيل الخادم.
3. **سجلات التهديدات والأحداث (Threats & Audits):**
   - جداول `threat_logs`, `blocked_ips`, `blocked_events`, `login_attempts`, `audit_logs`.
   - يخزن كل طلب خبيث مع تفاصيله (IP, Payload, Severity, ML Score).
4. **إدارة الحوادث (Incident Management):**
   - جداول `incidents` و `incident_actions`.
   - **العلاقة:** (1-to-M) لتتبع دورة حياة الحادث الأمني (مفتوح، قيد المعالجة، مغلق) وتسجيل من قام بتغيير الحالة.
5. **تتبع الذكاء الاصطناعي والمساعد الذكي:**
   - جداول `ml_model_runs` (لتسجيل دقة النموذج وتاريخ تدريبه) و `chatbot_sessions` للمساعد الذكي.

---

## 4. رحلة المستخدم (User Flow)
تم تصميم النظام لخدمة سيناريوهين رئيسيين:

**السيناريو الأول: رحلة الطلب الخارجي (Client Request Flow)**
1. يرسل العميل طلباً (مثلاً: محاولة تسجيل دخول أو بحث عن منتج).
2. يمر الطلب عبر الـ **Security Manager**:
   - التحقق من الـ IP في قائمة الحظر (`blocked_ips`).
   - فحص هجمات القوة الغاشمة (Brute Force) وسرعة الطلبات (Rate Limiting).
   - فحص (CSRF / SSRF).
   - تحليل محتوى الطلب (Layer 1: WAF Rules).
   - تحليل محتوى الطلب (Layer 2: ML Model).
3. **القرار:** إذا تم رصد تهديد، يتم حظر الطلب وإرسال تنبيه للوحة التحكم (Dashboard). إذا كان آمناً، يتم تنفيذ العملية بنجاح.

**السيناريو الثاني: رحلة محلل الأمن السيبراني (Analyst Flow)**
1. يسجل المحلل دخوله للوحة التحكم ويحصل على توثيق مشفر (JWT).
2. يشاهد لوحة تحكم تفاعلية تعرض الهجمات المباشرة (Live Threats).
3. يدخل لصفحة **الحوادث (Incidents)** لمراجعة التنبيهات عالية الخطورة.
4. يستخدم **المساعد الذكي (Chatbot)** المدمج للاستفسار عن طبيعة هجمة معينة أو طلب اقتراحات للتخفيف من حدتها.

---

## 5. الميزات المبتكرة (Innovative Features)
1. **محرك الفحص الهجين (Hybrid Detection):** دمج سرعة القواعد الثابتة مع ذكاء نماذج تعلم الآلة لاكتشاف الهجمات غير المعروفة، مع ميزة التقييم (Risk Score) للتقليل من الإيجابيات الخاطئة.
2. **أتمتة الاستجابة (Automated Mitigation):** حظر تلقائي (Auto-blocking) لعناوين IP التي تتجاوز حد محاولات تسجيل الدخول الفاشلة أو التي ترسل حمولات خبيثة شديدة الخطورة (Critical).
3. **مساعد أمني ذكي (AI Chatbot):** دمج مساعد آلي مخصص للمحللين الأمنيين داخل لوحة التحكم لتقديم تفسيرات فورية للهجمات أو تقديم توصيات لإصلاح الثغرات.
4. **تحديث القواعد ديناميكياً:** إمكانية إضافة أو تعطيل قواعد الحماية (Regex Rules) مباشرة من قاعدة البيانات دون الحاجة لإعادة نشر الكود المصدري.

## 6. الخلاصة والأسئلة المتوقعة من لجنة التحكيم

**س1: كيف يتعامل النظام مع "الإيجابيات الخاطئة" (False Positives) الصادرة من نموذج الذكاء الاصطناعي بحيث لا يعطل عمل المستخدمين الشرعيين؟**
* **الإجابة:** يعتمد النظام آلية تصنيف متدرجة؛ إذا كانت نسبة ثقة النموذج (Risk Score) عالية جداً يتم حظر الطلب فوراً. أما إذا كانت النسبة متوسطة، يتم تمرير الطلب ولكنه يُسجل تحت حالة (Monitor) في لوحة التحكم. يمكن للمحلل مراجعة هذه الحالات لاحقاً وإعادة تدريب النموذج (Retrain) لتحسين الدقة مستقبلاً.

**س2: لماذا تم استخدام SQLite بدلاً من قواعد بيانات أضخم مثل PostgreSQL أو SQL Server، وهل ستتحمل ضغط الطلبات؟**
* **الإجابة:** تم اختيار SQLite لتبسيط عملية النشر كنسخة أولية (PoC). ومع ذلك، تم تفعيل خاصية (WAL - Write-Ahead Logging) وتقليل قفل قاعدة البيانات، مما يسمح بعمليات القراءة والكتابة المتزامنة بكفاءة بين واجهة الـ API ولوحة التحكم. خطة التطوير المستقبلية تتضمن الترقية إلى SQL Server بسلاسة بفضل هيكلة الكود النظيفة.

**س3: ما هي الآلية المستخدمة لحماية لوحة التحكم (SIEM Dashboard) نفسها من الاختراق؟**
* **الإجابة:** لوحة التحكم مفصولة تماماً عن الـ API الأساسي (تعمل على منفذ مختلف). وتُحمى عبر نظام مصادقة يعتمد على (JWT - JSON Web Tokens) مع تشفير كلمات المرور (Hashing). كما يتم تطبيق صلاحيات (RBAC) لضمان عدم وصول المستخدمين العاديين لبيانات حساسة، بالإضافة لتضمين ترويسات حماية متقدمة (Security Headers) مثل X-Frame-Options و CSP لمنع هجمات الـ Clickjacking.
