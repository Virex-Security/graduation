# العرض التقديمي الشامل: نظام Virex الأمني
**"حماية ذكية لواجهات برمجة التطبيقات مع مراقبة وتحليل في الوقت الفعلي"**

---

## 1. الفكرة والدافع (The Vision)
مع التحول السريع نحو الخدمات السحابية والاعتماد الكبير على واجهات برمجة التطبيقات (APIs)، أصبحت هذه الواجهات الهدف الأول للمخترقين. جدران الحماية التقليدية (Traditional WAFs) لم تعد كافية لأنها تعتمد على "قواعد ثابتة" يسهل تجاوزها.
من هنا جاءت فكرة **Virex**: نظام حماية متقدم يجمع بين الصرامة الكلاسيكية والذكاء الاصطناعي ليكون بمثابة حارس أمن ومحلل بيانات في نفس الوقت. نحن لا نمنع الهجمات فحسب، بل نقدم رؤية واضحة (Visibility) لما يحدث داخل الشبكة من خلال لوحة تحكم SIEM (Security Information and Event Management).

---

## 2. المعمارية الهندسية (System Architecture)
تم بناء النظام بأسلوب يضمن الأداء العالي والفصل المنطقي للمسؤوليات (Separation of Concerns):

* **وحدة الحماية (Security API Service):**
  * مبنية باستخدام `Flask` وتعمل كبوابة عبور (Gateway) لكل الطلبات.
  * تتولى مسؤولية فحص الطلبات الواردة قبل وصولها للمخدم الداخلي.
* **وحدة المراقبة (SIEM Dashboard):**
  * خدمة منفصلة تماماً (تعمل على منفذ مختلف `8070`) تضمن عدم تأثر المراقبة بضغط الطلبات على الـ API.
  * توفر واجهة تفاعلية مبنية بتقنيات (HTML/CSS/JS) مع قوالب Flask.
* **وحدة تخزين البيانات المتزامنة:**
  * نستخدم `SQLite` مع تفعيل نمط `WAL (Write-Ahead Logging)` الذي يسمح للـ API بالكتابة (تسجيل الهجمات) وللوحة التحكم بالقراءة (عرض البيانات) في نفس اللحظة دون حدوث قفل لقاعدة البيانات (Database Locks).

---

## 3. محرك الفحص الذكي متعدد الطبقات (Multi-layered Detection Engine)
هذا هو "العقل المدبر" للمشروع، حيث لا نعتمد على آلية واحدة، بل يمر الطلب بعدة فلاتر (Pipeline):

1. **فلترة النطاق الترددي والسلوك (Rate Limiting & Scanner Detection):** 
   * صد هجمات الحرمان من الخدمة (DDoS) وإيقاف روبوتات الفحص (Scanners) التي تبحث عن ملفات حساسة مثل `.env` أو `wp-admin`.
2. **الطبقة الكلاسيكية (Rule-based WAF):**
   * فحص سريع يعتمد على التعابير النمطية (Regex) المخزنة في قاعدة البيانات لاكتشاف الهجمات المعروفة فوراً (مثل SQL Injection و XSS).
3. **طبقة الذكاء الاصطناعي (ML Layer):**
   * للطلبات التي تتجاوز الطبقة الأولى، يتم تمريرها لنموذج تعلم آلي (Random Forest + TF-IDF) مدرب مسبقاً.
   * وظيفة هذه الطبقة هي اكتشاف الهجمات المعقدة أو المتخفية (Zero-day/Obfuscated payloads) بناءً على "فهم سياق النص" وليس مجرد البحث عن كلمات مفتاحية.

---

## 4. نظام الاستجابة وإدارة الحوادث (Incident Response & Management)
نظام Virex ليس مجرد "جدار ناري" سلبي، بل يمتلك نظام استجابة متكامل:

* **الحظر التلقائي (Auto-blocking):** إذا تم رصد محاولات تخمين كلمات مرور متكررة (Brute Force) أو هجمات شديدة الخطورة، يقوم النظام بوضع الـ IP في القائمة السوداء تلقائياً لفترة زمنية محددة.
* **تتبع دورة حياة الحادث (Incident Tracking):** أي هجوم خطير يتم تحويله إلى "تذكرة حادث أمني" (Incident Ticket) داخل لوحة التحكم، حيث يمكن لمحلل الأمن تغيير حالتها (مفتوح، قيد المعالجة، مغلق) وتسجيل ملاحظاته.
* **سجل التدقيق (Audit Trails):** كل إجراء يقوم به المستخدم داخل النظام (تسجيل دخول، تغيير إعدادات، تفعيل قاعدة حماية) يتم تسجيله لضمان المساءلة (Accountability).

---

## 5. ميزات القوة والإبداع في المشروع (Key Differentiators)
ما الذي يجعل مشروعنا مميزاً ومستعداً لبيئات العمل الحقيقية؟

* **تحديث القواعد أثناء التشغيل (Hot-swappable Rules):** يمكن لمدير النظام إضافة أو تعديل أو إيقاف قواعد الحماية (Regex) من لوحة التحكم مباشرة، وستنعكس على الفور في الـ API دون الحاجة لإيقاف الخادم أو إعادة تشغيل الكود.
* **المساعد السيبراني المدعوم بالـ NLP (Dobby Chatbot):** دمجنا مساعداً ذكياً داخل لوحة التحكم يساعد المحللين الأمنيين المبتدئين. يمكن للمحلل أن يسأل المساعد عن تفاصيل هجوم معين، وسيقوم الـ Chatbot بتحليل السياق وتقديم شرح وتوصيات للحل.
* **تحمل التزامن (Concurrency Handling):** استخدمنا خيوط المعالجة (Threading) في إرسال التنبيهات (Webhooks/Requests) من الـ API إلى الـ Dashboard لضمان ألا يؤدي تسجيل الهجوم إلى إبطاء استجابة الموقع للمستخدم الشرعي.

---

## 6. الجانب الأمني الخاص بالنظام نفسه (Securing the Security System)
لم نغفل عن حماية النظام الذي يحمي الأنظمة!
* **المصادقة الآمنة (Authentication):** لوحة التحكم محمية بنظام (JWT - JSON Web Tokens) مع تخزين الرمز في ملفات تعريف ارتباط محمية (HttpOnly Cookies) لمنع سرقتها عبر XSS.
* **إدارة الصلاحيات (RBAC):** النظام يدعم أدواراً مختلفة (Admin, Analyst, User). المحلل (Analyst) يمكنه فقط القراءة وتقديم التقارير، بينما الـ (Admin) هو الوحيد القادر على حظر الـ IPs أو تعديل قواعد الحماية.
* **حماية كلمات المرور:** يتم تخزين جميع كلمات المرور في قاعدة البيانات باستخدام خوارزميات تشفير قوية (Hashing) عبر مكتبة `werkzeug.security` ولا تحفظ كنصوص واضحة أبداً.

---

## 7. الخاتمة والرؤية المستقبلية (Future Roadmap)
النظام في نسخته الحالية يقدم إثبات مفهوم (Proof of Concept) قوي جداً لحماية واجهات الـ API. 
**خطط التطوير المستقبلية تتضمن:**
* دعم قواعد بيانات ضخمة وموزعة (مثل PostgreSQL أو ElasticSearch) للتعامل مع ملايين السجلات يومياً.
* استبدال آلية التحديثات الحالية (HTTP Requests) بـ WebSockets لعرض التنبيهات في لوحة التحكم بشكل لحظي (Real-time Streaming) بأقل استهلاك للموارد.
* إتاحة واجهة مستخدم (UI) لتدريب نموذج الذكاء الاصطناعي على بيانات مخصصة لكل عميل لزيادة الدقة وتقليل نسبة الإيجابيات الخاطئة (False Positives).
